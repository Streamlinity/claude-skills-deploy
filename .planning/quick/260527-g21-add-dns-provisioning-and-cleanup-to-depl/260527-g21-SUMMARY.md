# Phase 260527-g21 Plan 01: Add DNS Provisioning and Cleanup Summary

**One-liner:** Cloudflare DNS A record provisioning and teardown wired into validate/provision/e2e/cleanup via lib-dns-api.sh following lib-coolify-api.sh patterns.

## What Was Built

### Task 1: lib-dns-api.sh + validate.sh + template + schema.md

Created `scripts/lib-dns-api.sh` (369 lines) — Cloudflare-first DNS library following `lib-coolify-api.sh` exactly:

- `dns_load_credentials(yaml_path)` — parses `dns:` block via inline python3/pyyaml, resolves token from Doppler staging config or `coolify.json` field, exports `DNS_PROVIDER`, `DNS_ZONE_NAME`, `DNS_API_TOKEN`
- `dns_load_credentials_from_env()` — loads from already-set shell vars (used by cleanup-deployment.sh which has no yaml path available)
- `dns_check_credentials(yaml_path)` — read-only presence check returning 0/1 (used by validate.sh)
- `dns_cf_curl(method, path, [body])` — Cloudflare REST wrapper, same shape as `coolify_curl`
- `dns_cf_get_zone_id(zone_name)`, `dns_cf_find_record(zone_id, name, type)`, `dns_cf_upsert_a_record(zone_id, name, ip)`, `dns_cf_delete_record(zone_id, record_id)`
- `dns_upsert_a_record(fqdn, ip)` — provider dispatch, zone suffix validation, echoes `zone_id|record_id`
- `dns_delete_record(zone_id, record_id)` — provider dispatch
- `dns_verify_a_record(fqdn, expected_ip)` — round-trip GET confirmation, hard-fails on mismatch

Extended `init/templates/coolify.yaml.tmpl` with `dns:` block using `{{DNS_PROVIDER}}`, `{{DNS_ZONE_NAME}}`, `{{DNS_CREDENTIAL_SOURCE}}`, `{{DNS_CREDENTIAL_KEY}}` tokens with `# CHANGE:`/`# LEAVE:` annotations.

Extended `scripts/validate.sh` to source `lib-dns-api.sh`, parse the `dns:` block, check zone suffix against both staging and production domains, and call `dns_check_credentials`. Errors accumulate via the existing `fail()` pattern with `MISSING:DNS_CREDENTIAL:<key>` format.

Added `### Optional DNS block` section to `docs/schema.md` covering all four fields, examples, Cloudflare token scope requirements, Doppler vs coolify.json storage, and updated Field Lifecycle and Validation sections.

## Decisions Made

1. **VPS IP resolution strategy:** `vps_ip` field in `coolify.json` (cached, avoids SSH on re-runs) → SSH `ifconfig.me` fallback. Hard-fail if empty or not dotted-quad. This avoids a DNS lookup or Coolify API round-trip and keeps parity with the existing SSH host pattern.

2. **Credential storage (doppler vs coolify_json):** Both paths supported. Doppler is the default and recommended — the token is stored once in the staging config since DNS credentials are zone-level. `coolify_json` is the escape hatch for operators who prefer not to put a Cloudflare token in their app's Doppler project.

3. **DNS deletion unconditional in e2e trap:** Even when `--keep` is set (operator inspecting live Coolify state), DNS records are always cleaned up. Orphaned DNS records in a DNS zone are harder to audit than orphaned Coolify apps — the asymmetry is documented in a comment in `cleanup()`.

4. **DNS record IDs not written to coolify.yaml:** `provision.sh` re-derives record IDs from upsert on every run (idempotent lookup-then-patch). The IDs flow to the e2e report only. Persisting them in a committed file would leak zone-specific state.

5. **TSV separator:** ASCII record separator (0x1E) used between DNS record entries in the report TSV field to avoid shell word-splitting on newlines when using `IFS=` read patterns.

## Known Limitations

- **Cloudflare only:** No Route53, Hetzner, or Namecheap in this pass. Provider dispatch shim (`case "$DNS_PROVIDER"`) makes adding providers straightforward.
- **No AAAA/CNAME support:** Only A records. IPv6 (AAAA) and alias records (CNAME) are out of scope.
- **No Let's Encrypt wait:** `provision.sh` creates the A record and verifies it round-trips in the Cloudflare API, but does not poll for actual DNS propagation or HTTPS cert issuance. Operators may still need to wait 1–2 minutes before HTTPS is live.
- **vps_ip not auto-saved:** `provision.sh` resolves the IP via SSH but does not write `vps_ip` back to `coolify.json` (avoiding mutations to the credentials file). On each re-run without `vps_ip` cached, an SSH round-trip occurs.
- **dns_default in coolify.json not prompted by `/setup-coolify init`:** The `dns_default` sub-object (used by `e2e.sh` to inject DNS into test yaml) must be set manually in `~/.claude/coolify.json`. Not surfaced through any interactive prompt yet.
