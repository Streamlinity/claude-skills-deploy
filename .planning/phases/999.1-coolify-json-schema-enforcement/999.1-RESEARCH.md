## Summary

Phase 999.1 is a hardening phase with no new external dependencies — all work is changes to existing bash scripts, docs, and test fixtures. The codebase is mature and the patterns are well-established.

The three major work streams are: (1) **validate.sh surgery** — promoting `doppler_token` from warn to hard fail, adding Tier 2 cross-checks, adding an env_vars grep scan, adding the `.coolify/validate.sh` hook, and stripping the `--seed-from-env` mutation path entirely; (2) **new `seed` subcommand** — extracting the gap-fill logic from validate.sh into a standalone `scripts/seed.sh` with explicit invocation from both provision.sh and SKILL.md; (3) **docs + tests** — updating schema.md, SKILL.md, setup-guide.md, and adding contract tests for the new validation behaviors plus a fixture test for the hook.

**Primary recommendation:** Work stream by stream. Each can be an independent plan because they have clean interfaces (validate.sh is called by provision.sh and e2e.sh; seed.sh is a new file; tests are additive).

### Pattern 1: Error Accumulation in validate.sh

**What:** `ERRORS` counter + `fail()` function collects all errors before exit.

**Code (validate.sh lines 38-39):**
```bash
ERRORS=0
fail() { echo "FAIL: $*" >&2; ERRORS=$((ERRORS+1)); }
```
Then at the end: `[ "$ERRORS" -gt 0 ] && exit 1`.

**When to use:** All new Tier 1 and Tier 2 checks in validate.sh MUST use this pattern — do not `exit 1` inline inside checks.

### Pattern 2: Python Inline Heredoc for JSON/YAML Parsing

**What:** `python3 - "$arg" <<'PY' ... PY` for multi-field extraction.

**Example (lib-doppler-api.sh lines 23-29):**
```bash
read -r _acct _tok < <(python3 - "$COOLIFY_REGISTRY" "$server_alias" <<'PY'
import json, sys
d = json.load(open(sys.argv[1]))
s = d.get('servers', {}).get(sys.argv[2], {})
print(s.get('doppler_account', ''), s.get('doppler_token', ''))
PY
)
```

**For Tier 1 doppler_token check in validate.sh:** The existing warn block (lines 89-101) already reads `doppler_token` and `cloudflare_api_token` together. The Tier 1 change means moving `doppler_token` from the `WARNINGS` section to the `ERRORS` section — it's not a new code pattern, just a severity promotion.

### Pattern 3: Tier 2 Cross-Check Structure

**What:** Read a field from coolify.yaml, check the corresponding coolify.json field, fail if feature is on and field is absent.

**New code to add after the existing MSRV-07 deploy_server/deploy_ssh_host coupling check (validate.sh ~line 170):**
```bash
# Tier 2: cloudflare_api_token required when dns.credential_source: coolify_json
eval "$(python3 "$SCRIPT_DIR/lib-config.py" emit-dns-vars "$YAML_PATH")"
# cred_source is set by emit-dns-vars
if [ "${cred_source:-}" = "coolify_json" ]; then
  _cf_token=$(python3 "$SCRIPT_DIR/lib-config.py" get-json-field \
    "$HOME/.claude/coolify.json" "$SERVER" "${cred_key:-cloudflare_api_token}")
  if [ -z "$_cf_token" ]; then
    fail "INVALID:coolify.json:servers.$SERVER.${cred_key:-cloudflare_api_token} (required when dns.credential_source: coolify_json in coolify.yaml)"
  fi
fi
```

Note: `emit-dns-vars` is already called later in validate.sh (line 353). The cross-check needs to happen BEFORE the Doppler key checks so it's in the schema-enforcement section.

### Pattern 4: lib-config.py get-json-field

**What:** Safe extraction of a single field from coolify.json via the existing lib-config.py helper.

**Usage:**
```bash
VALUE=$(python3 "$SCRIPT_DIR/lib-config.py" get-json-field \
  "$HOME/.claude/coolify.json" "$SERVER" "field_name")
```
Returns empty string when field is absent (no error exit). Safe for optional field checks.

### Pattern 6: SKILL.md Subcommand Table Routing

**What:** SKILL.md is read by Claude (not executed). It is prose + tables. The `## Subcommands` table describes each form. The `## Execution flow` section describes what happens. Claude routes to the correct script based on `$ARGUMENTS`.

**Current subcommand table:**
| Form | Action |
|------|--------|

| `/setup-coolify plan` | Read-only diff |
| `/setup-coolify init_cicd` | Interactive setup |

| `/setup-coolify validate` | Validate (currently describes gap-fill side-effect) |

**New rows to add:** `seed` and `provision`. The `validate` description must be rewritten to say "strictly read-only — no Doppler mutations".

The `provision` subcommand routing: since provision.sh already exists and ignores `provision` as an arg (it passes arg through as YAML_PATH), the cleanest approach is to handle it in SKILL.md (Claude sees `provision` arg, runs `provision.sh` without it) OR strip it in provision.sh's arg parsing. The CONTEXT.md leaves this to Claude's discretion.

### Anti-Patterns to Avoid
- **Inline `exit 1` inside validate.sh checks:** Always increment `ERRORS` and fall through to the accumulated exit at the end. Exceptions: the few existing "early exit" cases where further checks would be meaningless (e.g., coolify.json server alias not found — nothing else can proceed).
- **Shell text munging for YAML:** Never use sed/awk to parse YAML. Use `python3 lib-config.py` or a heredoc `<<'PY'` block with `import yaml`.
- **Hardcoded server aliases in new scripts:** All scripts must be domain-agnostic; server alias is always read from coolify.yaml `server:` field.

### validate.sh Gap-Fill (the "side effect" to remove)

**Location:** validate.sh lines 233–316. The gap-fill logic lives in two functions:
- `_parse_env_file()` — parses KEY=VALUE from a .env file
- `_fill_missing_from_env()` — calls `doppler secrets upload` for each missing key

The `--seed-from-env` flag (line 24) currently gates mutation. The phase decision is:
1. Remove `--seed-from-env` flag entirely from validate.sh
2. Remove `_parse_env_file()` and `_fill_missing_from_env()` functions from validate.sh
3. Remove the `if [ "$SEED_FROM_ENV" = "true" ]` block entirely
4. Remove the hint "validate: .env file(s) found — run with --seed-from-env to fill missing Doppler keys"
5. Create `scripts/seed.sh` containing this logic (described below)

**Impact on provision.sh:** provision.sh calls `validate.sh` at line 32 without `--seed-from-env`. So removing the flag from validate.sh has no effect on provision.sh's validate call. The seed step needs to be added as a new step in provision.sh after validate passes, before any Coolify mutations.

**Exact snippet to change:** Lines 93–96 and 107–112 in validate.sh.

### env_vars Grep Scan

**What to build:** After the Tier 2 cross-checks in validate.sh, add a grep scan:
- For each key in `$ENV_VARS`: grep the working directory (excluding .git, node_modules, vendor) for the key name.
- If key is NOT found in codebase: emit advisory `WARN: env_vars: $KEY declared in coolify.yaml but not found in codebase (stale?)`
- For common patterns grep for: `process.env.KEY` (JS), `os.environ['KEY']` / `os.getenv('KEY')` (Python), `$KEY` / `${KEY}` (shell), just `KEY` as a literal match in non-binary files.
- For keys found in codebase: no action needed (they're referenced).

The CONTEXT.md real-world example: `ADMIN_PASSWORD` / `ADMIN_SECRET` declared in coolify.yaml but no longer used in the codebase — grep scan should output a WARN (not a FAIL).

**Best-effort grep pattern (Claude's discretion):** Use `grep -rl "$KEY" . --include="*.{js,ts,jsx,tsx,py,sh,env*,yaml,yml}"` as a language-agnostic scan. Exclude `.git/`, `node_modules/`, `vendor/`, `.next/`, `dist/`. WARN-only, never FAIL.

**Also scan for the reverse:** grep codebase for `process.env.` / `os.environ\[` / `os.getenv\(` and extract key names, then compare against env_vars list. This is more complex and may generate false positives; implement as best-effort WARN.

**Important:** The grep scan runs from the `YAML_DIR` (directory containing coolify.yaml), not `SCRIPT_DIR`. It must handle the case where validate.sh is called from a non-repo directory (the path may be an absolute path to a YAML in a temp dir during E2E testing) — skip the scan when codebase indicators aren't present.

## Common Pitfalls

### Pitfall 1: Doppler Token Export Timing

**What goes wrong:** `doppler_load_account` (lib-doppler-api.sh line 42) exports `DOPPLER_TOKEN` when `doppler_token` is present in coolify.json. But if validate.sh checks for `doppler_token` before calling `doppler_load_account`, the export hasn't happened yet — the Doppler CLI calls for `doppler_check_key` will use ambient auth.

**Current state:** validate.sh calls `coolify_load_server` at line 63 and `doppler_load_account` at line 68. These happen before the Doppler key checks. The Tier 1 doppler_token FAIL check should go between lines 68 and 80 (after `doppler_load_account` so the token is already exported even though the check fails).

**How to avoid:** Put the Tier 1 doppler_token check AFTER `doppler_load_account` is called so the existing export mechanism still runs even when we fail.

### Pitfall 2: emit-dns-vars Side Effects in validate.sh

**What goes wrong:** validate.sh calls `eval "$(python3 lib-config.py emit-dns-vars "$YAML_PATH")"` at line 353. This sets shell variables like `provider`, `zone_name`, `cred_source`, `cred_key`. If the Tier 2 cross-check also calls `emit-dns-vars` before line 353, the variables are set twice — harmless but redundant.

**How to avoid:** Call `emit-dns-vars` once, then do both the Tier 2 cross-check and the DNS validation with the same variables. Restructure the Tier 2 check to come after the first `emit-dns-vars` call.

**Alternative (simpler):** Add Tier 2 cross-check inline at the existing DNS block (line 353+), reusing the already-set `cred_source` and `cred_key` variables.

### Pitfall 3: grep scan in E2E context

**What goes wrong:** During E2E testing, validate.sh is called with `YAML_PATH` pointing to `$WORK_DIR/coolify.yaml`. The "codebase" relative to that YAML is the temp dir — no source files present. The env_vars grep scan will output WARNs for every key (`HELLO`, `E2E_TEST`) because they're not found in the temp dir.

**How to avoid:** Either (a) skip grep scan when `YAML_DIR` has no source files (check for `.git` dir or similar indicator), or (b) make the grep scan non-blocking (WARN only, never FAIL) — which is already the decision. The WARNs during E2E are acceptable and the test still passes.

**Simpler guard:** Check `[ -d "$YAML_DIR/.git" ] || skip grep scan with a note`. The CONTEXT.md says "best-effort coverage" so this is acceptable.

### Pitfall 4: Backward Compatibility of validate.sh SKILL.md description

**What goes wrong:** The current SKILL.md `## validate flow` section (lines 128–135) explicitly documents the gap-fill behavior as a feature: "Gap-fill from local .env files (automatic, targeted mutation)." This will be removed. The `init_cicd` section (line 80) also mentions gap-fill via `init_app` flow. Both must be updated.

**How to avoid:** After changing validate.sh, do a final pass over SKILL.md to remove all references to validate.sh mutations. The `## validate flow` section becomes a shorter pure-readonly description.

### Pitfall 5: provision.sh calls validate.sh first (no double-seeding)

**What goes wrong:** If provision.sh calls `seed.sh` and then `validate.sh`, and validate.sh still had gap-fill logic, keys would be seeded twice. With gap-fill removed from validate.sh, this is not an issue — but the ordering matters for correctness: seed AFTER validate passes (so we know the environment is sane before touching Doppler).

**How to avoid:** The CONTEXT.md decision is clear: provision orchestrates as `validate → seed (if .env files present) → mutate Coolify`. This ordering is correct.
