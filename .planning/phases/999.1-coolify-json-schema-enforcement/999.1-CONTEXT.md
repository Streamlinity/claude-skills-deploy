<decisions>
## Implementation Decisions

### Tier 1 (always required — hard fail)
- `url`, `api_key`, `doppler_account`, `ssh_host` — already enforced
- `doppler_token` — promote to Tier 1 hard fail (currently non-blocking warn); without it Doppler CLI silently targets wrong workspace in multi-workspace setups

### Tier 2 (feature-gated — required when feature is enabled in coolify.yaml)
- `cloudflare_api_token` → required when `dns.credential_source: coolify_json` in coolify.yaml; cross-check validate.sh against coolify.yaml at validation time
- `deploy_ssh_host` → required when `deploy_server:` is set in coolify.yaml
- `deploy_vps_ip` → required when `deploy_server:` set AND dns is enabled
- These are NOT required for repos that don't use those features

### Tier 3 (truly optional — working defaults, no functional gap)
- `server_name` — defaults to "localhost" (correct for standard Coolify)
- `vps_ip` — resolved via SSH ifconfig.me if absent
- `dns_default` — only for E2E DNS testing
- No warnings for missing Tier 3 fields

### validate.sh changes
- Strictly read-only: remove the Doppler gap-fill side-effect entirely from validate
- Add Tier 1 check for `doppler_token` (hard fail)
- Add Tier 2 cross-checks against coolify.yaml (hard fail when feature is on and field is absent)
- Add env_vars grep scan: flag vars in env_vars list not found in codebase (stale); flag vars grep-found in codebase but not in coolify.yaml (missing). Language-agnostic grep — not 100% coverage but catches obvious cases.
- Support opt-in `.coolify/validate.sh` hook in target repo: called with coolify.yaml path as $1, exits non-zero blocks provision. Claude calls it if present, skips if absent.

### New subcommands
- `seed` — explicit Doppler gap-fill from .env.local (seeds stg + dev) and .env.production (seeds prd). Logs every key set. Never overwrites existing Doppler values.
- `provision` — explicit alias for blank (current default behavior unchanged). Makes subcommand table complete.
- Keep blank as alias for `provision` for backward compatibility.
- provision orchestrates: validate → seed (if .env files present) → mutate Coolify → write UUIDs. Does NOT auto-call plan (plan is opt-in, provision logs EXISTS/CREATE/PATCH inline).

### Examples file
- `examples/coolify.json.example` with REPLACE_THIS placeholders for all Tier 1 fields
- doppler_token included as Tier 1 field (not "optional" section)
- Tier 2 fields in annotated optional block with "ONLY if..." comments
- Tier 3 fields documented but clearly marked optional with defaults shown
- Pattern matches examples/skillmap/coolify.yaml style

### Docs
- docs/schema.md: add explicit 3-tier model section, update Required Fields table to include doppler_token, mark Tier 2 fields as conditionally required with coolify.yaml cross-reference
- SKILL.md: update subcommand table — add seed, provision rows; update validate description (read-only, no gap-fill); update plan description
- docs/setup-guide.md: update any references to validate side-effects or subcommand names

### E2E and tests
- Run hello-world E2E deployment (test/e2e.sh) with E2E_SERVER=vultr-stream
- Add contract checks or unit tests for: Tier 1 doppler_token check, Tier 2 cross-check logic, env_vars grep scan behavior
- Add fixture test verifying .coolify/validate.sh hook is called when present
- Update test/e2e.sh if validate/seed/provision invocation changes affect it

### Claude's Discretion
- Exact grep pattern for env_vars scan (handle .env prefix stripping, process.env patterns for JS, os.environ for Python, etc. — best-effort coverage)
- Error message wording for Tier 2 cross-check failures (should name both files)
- Whether `provision` subcommand is handled in SKILL.md routing or in provision.sh arg parsing
</decisions>

### Established patterns
- validate.sh uses error accumulation (`ERRORS` counter + `fail()` function) before exit — preserve this
- Python inline `python3 -c` / heredoc `<<'PY'` for YAML parsing — use for coolify.yaml cross-checks
- `set -euo pipefail` in all scripts
- lib files are sourced, not executed — changes to lib-doppler-api.sh affect all callers

### Doppler token handling
- `lib-doppler-api.sh` already reads `doppler_account` from coolify.json
- `doppler_token` needs to be exported as `DOPPLER_TOKEN` env var before every Doppler CLI call if present
- Pattern: `export DOPPLER_TOKEN=$(python3 ... read from coolify.json)` at top of provision/validate
