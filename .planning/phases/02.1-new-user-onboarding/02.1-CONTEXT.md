<decisions>
## Implementation Decisions

### E2E defaults (test/e2e.sh lines 44-49)

- **D-01:** Remove the hardcoded defaults for `E2E_SERVER` and `E2E_BASE_DOMAIN`. If either is unset when the script runs, print a clear error message and exit 1. Do NOT silently use a maintainer-specific value.
- **D-02:** Error message format — name the missing var, explain conceptually what it refers to (an alias key in `~/.claude/coolify.json` that points to the user's Coolify instance — NOT a raw hostname), and point them to `/setup-coolify init` as the path to create one if they haven't. Example:
  ```
  ERROR: E2E_SERVER is required.
    Set it to the server alias key in ~/.claude/coolify.json (e.g. my-server).
    This alias points to your Coolify instance URL, API key, and SSH host.
    Run /setup-coolify init in any Claude Code session to create or extend ~/.claude/coolify.json.

    E2E_SERVER=my-server bash test/e2e.sh
  ```
  Apply the same pattern to `E2E_BASE_DOMAIN`, explaining it's the base domain under which test app URLs are constructed (e.g., `ci.example.com` produces `<project>-staging.ci.example.com`).
- **D-03:** `E2E_IMAGE` (line 49) keeps its default `ghcr.io/streamlinity/csd-hello-world:latest`. The image is public and domain-neutral. Add a comment block explaining: this is the maintainer's public test fixture image, it serves `/api/health → 200` and a known sentinel on `/`, and point users to `test/hello-world/` if they want to build and push their own test image.

### SKILL.md accuracy

- **D-04:** Remove the `coolify_get_github_app_uuid` reference from step 2. The function is dead code — `provision.sh` never calls it. Rewrite step 2 to accurately describe the lookup flow: `coolify_upsert_project`, `coolify_get_server_uuid` (from `coolify.json` server_name field, default `localhost`), `coolify_get_destination_uuid`.
- **D-05:** Rewrite step 6. `provision.sh` does NOT call `coolify_deploy_app`. The first deploy is triggered by the GitHub Actions push (the generated `deploy.yml`). Step 6 should say: "Write back `coolify_app_ids` to `coolify.yaml`. First deploy is triggered by pushing to `main`, which activates the generated `.github/workflows/deploy.yml`."
- **D-06:** Replace all maintainer-specific examples in the `init flow` section with generic placeholders:
  - `vultr-stream` → `my-server`
  - `https://coolify.cicd.streamlinity.com` → `https://coolify.example.com`
  - `streamlinity` → `my-doppler-account`
- **D-07:** Fix the broken "See also" reference: `.planning/codebase/COOLIFY_YAML_SCHEMA.md` does not exist. Replace with the correct path: `docs/schema.md`.

### README quick-start

- **D-08:** Add a "Quick start" section to `README.md` positioned above the existing "Prerequisites" section. Content: 5 commands in order, each with a one-line description of what it does and what it produces. Structure:
  1. `git clone ... ~/.claude/skills/setup-coolify` — installs the skill
  2. `/setup-coolify init` — interactive setup of `~/.claude/coolify.json` for your Coolify server
  3. `bash ~/.claude/skills/setup-coolify/init/init.sh` — bootstrap a new repo (writes `coolify.yaml` + `deploy.yml`)
  4. `/setup-coolify validate` — dry-run check (no mutations)
  5. `/setup-coolify` — provision Coolify apps and wire Doppler
  Link out to `docs/setup-guide.md` for full VPS setup prerequisites and to `docs/fork-guide.md` for adding a second domain to an existing setup.

### references/api-reference.md

- **D-09:** Replace all occurrences of `coolify.cicd.streamlinity.com` (and any other domain-specific values like `streamlinity`) with generic placeholders: `<your-coolify-domain>` for the Coolify instance URL, `<your-doppler-account>` for Doppler account slug. Add a one-line note at the very top of the file:
  ```
  > Replace `<your-coolify-domain>` with your Coolify instance root URL and `<your-doppler-account>` with your Doppler account slug throughout this document.
  ```

### Claude's Discretion

- Exact wording and formatting of the README quick-start section (as long as it covers the 5 steps and links)
- Whether to validate `E2E_SERVER`/`E2E_BASE_DOMAIN` at the top of the script before any other operations, or at the point of first use
- Whether to accumulate both missing-var errors before exiting, or report the first missing var and stop

</decisions>
